2. Data model & database schema
2.1 Database goals

The database must support:

defining ICPs
discovering companies
enriching companies
finding contacts
validating contacts
detecting signals
scoring leads
generating drafts
preventing duplicates
tracking pipeline runs
learning from campaign feedback
later expanding from single-tenant to multi-project

For now, use SQLite.

2.4 Tables in detail
1. projects

Even if single-tenant now, create this from day one.

CREATE TABLE projects (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    name TEXT NOT NULL,
    description TEXT,
    status TEXT DEFAULT 'active',
    created_at TEXT DEFAULT CURRENT_TIMESTAMP,
    updated_at TEXT DEFAULT CURRENT_TIMESTAMP
);

Example:

ActorDo outbound
Vibetrace ecommerce outbound
2. icps

Stores each ideal customer profile.

CREATE TABLE icps (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    project_id INTEGER NOT NULL,

    name TEXT NOT NULL,
    description TEXT,

    target_industries TEXT,
    target_geographies TEXT,
    target_company_sizes TEXT,
    target_revenue_range TEXT,

    target_personas TEXT,
    pain_points TEXT,
    competitors TEXT,
    buying_signals TEXT,
    exclusion_rules TEXT,

    offer_summary TEXT,
    value_proposition TEXT,
    outreach_angle TEXT,

    status TEXT DEFAULT 'active',

    created_at TEXT DEFAULT CURRENT_TIMESTAMP,
    updated_at TEXT DEFAULT CURRENT_TIMESTAMP,

    FOREIGN KEY (project_id) REFERENCES projects(id)
);

Use JSON strings for fields like:

["ecommerce", "DTC", "fashion retail"]

SQLite can store JSON as TEXT.

3. companies

Master company table.

One company should exist only once per domain.

CREATE TABLE companies (
    id INTEGER PRIMARY KEY AUTOINCREMENT,

    name TEXT,
    domain TEXT UNIQUE,
    website_url TEXT,

    linkedin_url TEXT,
    country TEXT,
    city TEXT,
    industry TEXT,
    description TEXT,

    employee_count INTEGER,
    revenue_estimate TEXT,

    ecommerce_platform TEXT,
    tech_stack TEXT,

    status TEXT DEFAULT 'new',

    created_at TEXT DEFAULT CURRENT_TIMESTAMP,
    updated_at TEXT DEFAULT CURRENT_TIMESTAMP
);

Recommended statuses:

new
discovered
enriched
qualified
rejected
suppressed

Important rule:

domain = main deduplication key

10. lead_candidates

This is the central table.

It connects:

ICP + company + contact

A company/contact can be a good lead for one ICP but not another.

CREATE TABLE lead_candidates (
    id INTEGER PRIMARY KEY AUTOINCREMENT,

    project_id INTEGER NOT NULL,
    icp_id INTEGER NOT NULL,
    company_id INTEGER NOT NULL,
    contact_id INTEGER,

    lead_status TEXT DEFAULT 'new',

    icp_fit_score REAL,
    signal_score REAL,
    persona_fit_score REAL,
    data_quality_score REAL,
    final_score REAL,

    qualification_summary TEXT,
    rejection_reason TEXT,

    ready_for_outreach INTEGER DEFAULT 0,

    created_at TEXT DEFAULT CURRENT_TIMESTAMP,
    updated_at TEXT DEFAULT CURRENT_TIMESTAMP,

    FOREIGN KEY (project_id) REFERENCES projects(id),
    FOREIGN KEY (icp_id) REFERENCES icps(id),
    FOREIGN KEY (company_id) REFERENCES companies(id),
    FOREIGN KEY (contact_id) REFERENCES contacts(id)
);

Recommended statuses:

2.5 Recommended indexes

These matter a lot.

CREATE INDEX idx_companies_domain ON companies(domain);
CREATE INDEX idx_contacts_email ON contacts(email);
CREATE INDEX idx_contacts_company_id ON contacts(company_id);
CREATE INDEX idx_lead_candidates_icp_id ON lead_candidates(icp_id);
CREATE INDEX idx_lead_candidates_company_contact ON lead_candidates(company_id, contact_id);
CREATE INDEX idx_signals_company_id ON signals(company_id);
CREATE INDEX idx_signals_icp_id ON signals(icp_id);
CREATE INDEX idx_email_drafts_lead_id ON email_drafts(lead_candidate_id);
CREATE INDEX idx_campaign_events_lead_id ON campaign_events(lead_candidate_id);
CREATE INDEX idx_suppression_value ON suppression_list(value);
CREATE INDEX idx_pipeline_runs_project_id ON pipeline_runs(project_id);
2.6 Important uniqueness rules
Companies
domain must be unique

Do not use company name as the main key.

Bad:

Acme Inc
ACME
Acme GmbH

Good:

acme.com
Contacts

Eventually add unique logic:

email unique if known
linkedin_url unique if known
company_id + full_name + job_title as fallback

SQLite partial uniqueness is possible, but keep it simple in MVP.

Lead candidates

Avoid creating the same lead twice:

CREATE UNIQUE INDEX idx_unique_lead_candidate
ON lead_candidates(icp_id, company_id, contact_id);

If contact_id is nullable, handle company-only leads in code.

2.8 Lead readiness logic

A lead should become ready_for_outreach = 1 only when:

company exists
company has domain
company is not suppressed
contact exists
contact has valid or high-confidence email
contact matches persona
at least one signal exists
final_score >= threshold
draft exists
draft passes quality check

Example threshold:

final_score >= 70
quality_score >= 70
email_status in valid, catch_all, risky
2.9 Scoring model fields

Recommended scoring formula:

final_score =
  30% ICP fit
  25% signal strength
  20% persona fit
  15% data quality
  10% freshness

Store the components in lead_candidates.

Do not only store final score. You need the breakdown to debug why a lead was selected.
