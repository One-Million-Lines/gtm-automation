9. Signal extraction module
9.1 Purpose

Detect and store why a company/contact is worth contacting now.

This module turns raw data into outreach intelligence.

Example:

Company uses Klaviyo
Company is hiring a CRM manager
Company recently migrated to Shopify Plus
Contact posted about email automation
Company has poor reviews about marketing tools

A lead without a signal is just a cold contact.
A lead with a verified signal becomes a reason-based outreach opportunity.

9.2 Main requirement

The module must support multiple sources for the same signal.

Example:

Signal: Uses Klaviyo

Source 1: BuiltWith detected Klaviyo
Source 2: Website HTML contains klaviyo.com
Source 3: Job post says “experience with Klaviyo required”
Source 4: Case study mentions Klaviyo

The system should not create 4 separate weak signals only.
It should create:

1 canonical signal
+ multiple supporting sources/evidence records
9.3 Core concept

Separate:

Signal

from:

Signal Evidence

Recommended new table:

CREATE TABLE signal_evidence (
    id INTEGER PRIMARY KEY AUTOINCREMENT,

    signal_id INTEGER NOT NULL,

    source_type TEXT NOT NULL,
    source_name TEXT,
    source_url TEXT,

    evidence_text TEXT,
    raw_data TEXT,

    confidence_score REAL,
    extracted_at TEXT DEFAULT CURRENT_TIMESTAMP,

    FOREIGN KEY (signal_id) REFERENCES signals(id)
);

Why this matters:

signals = the claim
signal_evidence = proof supporting the claim
9.4 Updated signal model

Current signals table should represent the canonical signal.

CREATE TABLE signals (
    id INTEGER PRIMARY KEY AUTOINCREMENT,

    company_id INTEGER,
    contact_id INTEGER,
    icp_id INTEGER,

    signal_type TEXT NOT NULL,
    signal_name TEXT NOT NULL,

    description TEXT,

    strength_score REAL,
    confidence_score REAL,

    status TEXT DEFAULT 'active',

    first_detected_at TEXT DEFAULT CURRENT_TIMESTAMP,
    last_detected_at TEXT DEFAULT CURRENT_TIMESTAMP,

    detected_by TEXT,
    raw_data TEXT,

    created_at TEXT DEFAULT CURRENT_TIMESTAMP,
    updated_at TEXT DEFAULT CURRENT_TIMESTAMP,

    FOREIGN KEY (company_id) REFERENCES companies(id),
    FOREIGN KEY (contact_id) REFERENCES contacts(id),
    FOREIGN KEY (icp_id) REFERENCES icps(id)
);

You can keep source_url and extracted_text for MVP, but better to move them into signal_evidence.

9.13 Canonicalization

Different sources may describe the same thing differently.

Examples:

klaviyo
Klaviyo tracking script
klaviyo.com
Klaviyo email platform

All should map to:

signal_type = competitor_usage
signal_name = Uses Klaviyo

Create a signal canonicalizer.

class SignalCanonicalizer:
    def canonicalize(self, candidate: dict) -> dict:
        pass

Example mapping:

COMPETITOR_SIGNAL_MAP = {
    "klaviyo": "Uses Klaviyo",
    "klaviyo.com": "Uses Klaviyo",
    "klaviyo tracking": "Uses Klaviyo",
    "omnisend": "Uses Omnisend",
    "mailchimp": "Uses Mailchimp"
}
9.14 Signal deduplication

Deduplicate by:

company_id
contact_id
icp_id
signal_type
canonical signal_name

If signal exists:

update last_detected_at
recalculate confidence
add new signal_evidence

If signal does not exist:

create signal
create signal_evidence

Recommended unique index:

CREATE UNIQUE INDEX idx_unique_signal
ON signals(company_id, contact_id, icp_id, signal_type, signal_name);

Note: SQLite treats NULLs as distinct, so handle contact_id carefully in code.

9.15 Evidence deduplication

Avoid duplicate evidence rows.

Deduplicate by:

signal_id
source_type
source_url
evidence_text hash

Add:

ALTER TABLE signal_evidence ADD COLUMN evidence_hash TEXT;
CREATE INDEX idx_signal_evidence_hash ON signal_evidence(evidence_hash);

Generate hash:

hash(source_type + source_url + normalized_evidence_text)
9.16 Confidence aggregation

When multiple sources support the same signal, confidence should increase.

Simple formula:

confidence = 1 - product(1 - evidence_confidence)

Example:

Evidence 1 confidence: 0.8
Evidence 2 confidence: 0.7

Combined:
1 - (1 - 0.8) * (1 - 0.7)
= 0.94

Cap at:

0.98

This prevents fake certainty.

9.17 Strength vs confidence

Keep these separate.

Confidence
How sure are we this signal is true?

Example:

BuiltWith says Klaviyo detected → high confidence
Random blog mention → medium confidence
Strength
How useful is this signal for outreach?

Example:

Uses Klaviyo → useful
Hiring Head of CRM → very useful
Mentions marketing on website → weak

A signal can be:

high confidence, low strength

Example:

Company has Google Analytics

or:

medium confidence, high strength

Example:

Review complains about Klaviyo pricing
9.18 Signal score per lead

For each lead candidate:

signal_score = weighted sum of relevant signals

Example weights:

competitor_usage: 25
hiring_intent: 30
pain_signal: 35
growth_signal: 15
tech_stack: 10
contact_activity: 20
manual_signal: 40

Then apply confidence:

effective_signal_score = strength_score * confidence_score * weight

Normalize to:

0–100
9.19 Multi-source score boost

Give extra weight for corroboration.

Example:

1 source → no boost
2 sources → +10%
3 sources → +20%
4+ sources → +30%

But cap final signal contribution.

Reason:

A signal confirmed by website + tech stack + job post is stronger than one source alone.
9.20 Source reliability weights

Not all sources are equal.

Example:

manual_note: 1.0
builtwith: 0.95
website_html: 0.9
company_careers_page: 0.9
job_board: 0.85
linkedin_company: 0.8
news: 0.75
review_site: 0.7
reddit: 0.5
ai_inference_only: 0.4

Final evidence confidence:

provider_confidence * source_reliability

9.22 LLM-based extraction prompt

Use AI for messy text sources.

Prompt:

You extract B2B outreach signals from source text.

Given:
- ICP definition
- company profile
- contact profile if available
- source text
- allowed signal types

Return JSON only:
{
  "signals": [
    {
      "signal_type": "",
      "signal_name": "",
      "description": "",
      "evidence_text": "",
      "strength_score": 0.0,
      "confidence_score": 0.0
    }
  ]
}

Rules:
- Use only the provided source text.
- Do not invent facts.
- Every signal must include exact evidence_text.
- If evidence is weak, lower confidence.
- If no signal exists, return {"signals": []}.
- Keep signal_name short and canonical.
9.23 Signal quality rules

Reject signals if:

no evidence text
no source type
no company_id and no contact_id
confidence_score < 0.4
signal is generic
signal is not relevant to ICP

Weak examples to reject:

Company has a website
Company uses Google Analytics
Company posts on LinkedIn
Company sells products

Useful examples:

Company uses Klaviyo
Company is hiring CRM Manager
Company mentions retention marketing
Contact owns lifecycle marketing
Company complains about high email platform cost

manual note: no expiry unless configured
9.26 Freshness scoring

Add freshness decay.

Example:

0–30 days old: 100%
31–60 days: 75%
61–90 days: 50%
90+ days: 25%

For job posts:

0–14 days: 100%
15–30 days: 70%
31+ days: stale

9.32 Definition of done

This module is complete when:

Signals are stored separately from evidence
One signal can have multiple sources
Duplicate evidence is avoided
Confidence increases when sources agree
Signal strength stays separate from confidence
Signals are linked to company/contact/ICP
Signals can be manually added
Signals can be rejected or marked stale
Lead signal score can be calculated from signals
