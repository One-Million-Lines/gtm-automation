Part 2 — Project review: scope, status, gaps to full autonomy
2.1 Project scope (recap)
Self-hosted GTM/outbound automation system. End-to-end flow:

ICP definition → company discovery & enrichment → contact discovery & enrichment → signal extraction → lead scoring → outreach generation → quality checks → send queue → reply ingestion & classification → multi-turn conversation drafting → experiments → exports → feedback → weight tuning

Multi-tenant (users + project memberships + roles), gated by JWT, deployable via Docker or systemd.

2.2 Implemented modules — inventory
Backend API routers (all behind JWT except /health and /auth):
auth, projects, icps, companies, contacts, enrichment, contact_enrichment, signals, leads, outreach, quality, sends, replies, conversations, experiments, exports, suppression, feedback, tuning, metrics, pipeline, orchestration, health — 23 routers, 133 routes.

Pipeline modules (composable steps in pipeline/modules/):
company_discovery, company_enrichment, contact_discovery, contact_enrichment, signal_extraction, lead_scoring, outreach_generation, quality_control, send, reply_tracking, multi_turn_drafter, suppression, experiment_scoring, engagement_metrics, feedback_ingestion, weight_tuning, export, full_pipeline — 18 modules.

Service layer (services/): 31 services covering ICP, scoring, quality, sends, replies, classifier, signal provider (HTTP-real + fake), email sender (SMTP/SendGrid/Postmark/fake), reply ingestor (IMAP + webhook + fake), LLM factory, orchestrator with cron evaluator, weight tuner, etc.

Frontend pages (frontend/src/pages/): 25 pages — Dashboard, ICPs, Companies, Contacts, Leads, LeadInbox, Outreach, Quality, Sends, Replies, DecisionTraces, Experiments+Detail, Exports+Detail, Feedback, Suppression, Templates, Schedules, Tuning, PipelineRuns, PipelineDashboard, Login/Register, Placeholder.

Auth & ops: bcrypt+JWT, project-scoped membership, audit log, rate limiter, structured JSON logging, optional Sentry. Docker, docker-compose, host nginx, in-container nginx, systemd unit, deploy script, full readme.

2.4 What's actually blocking "set it and forget it"
These are the only items still requiring a human to touch the system on a regular cycle (excluding the deliberate approval step):

Scheduler daemon. Add either:
a tiny scheduler_worker.py that loops time.sleep(60) → calls orchestrator_service.scheduler_tick(repos, runner), plus a systemd unit gtm-scheduler.service; or
a host crontab line: * * * * * curl -s -X POST -H "Authorization: Bearer $SVC_TOKEN" http://127.0.0.1:5214/pipeline/scheduler/tick.
This is the single biggest gap — the orchestration is built but unattended.
Wire reply pollers in startup.py. Mirror the email_sender pattern: read REPLY_INGESTOR=imap|webhook|fake env; if imap, install IMAPReplyIngestor and have the scheduler daemon call reply_service.poll(...) every N minutes.
Real prospecting connectors. Pick one of Apollo / Clearbit / Hunter / Dropcontact, implement a CompanySource and a ContactSource adapter, and add them as pipeline-module configs. Without this, you must keep importing CSVs manually.
Bounce / complaint webhook. Add /sends/webhook/{provider} for SendGrid/Postmark/SES events → mark sends bounced|complained, auto-add to suppression.
Auto-promote A/B winners. Add a small loop in experiment_service (called by scheduler) that flips the chosen variant to is_default=true once statistical thresholds are met.
Auto-rotate scoring weights. Schedule weight_tuner_service weekly; require an "apply" flag if you want a human gate.
