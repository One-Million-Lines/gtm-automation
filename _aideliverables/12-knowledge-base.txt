12. Knowledge base module
12.1 Purpose

The knowledge base stores everything the system learns over time.

It is the memory layer for:

campaign learnings
reply patterns
winning messages
objections
competitor mentions
call transcripts
manual notes
ICP refinements
positioning insights

Without this module, every campaign starts from zero.

With it, the system can make:

Campaign 2 smarter than Campaign 1
Campaign 3 smarter than Campaign 2
12.2 Main goal

Turn raw feedback into reusable outbound intelligence.

Examples:

People using Klaviyo complain about pricing
Founders respond better to cost-saving angles
CRM managers respond better to workflow automation angles
“Not looking right now” usually means bad timing, not bad ICP
Companies hiring lifecycle marketers are high-intent

12.6 Database table

Current table:

CREATE TABLE knowledge_items (
    id INTEGER PRIMARY KEY AUTOINCREMENT,

    project_id INTEGER NOT NULL,
    icp_id INTEGER,

    item_type TEXT NOT NULL,
    title TEXT,

    content TEXT,
    summary TEXT,

    source_type TEXT,
    source_url TEXT,

    tags TEXT,
    importance_score REAL,

    created_at TEXT DEFAULT CURRENT_TIMESTAMP,
    updated_at TEXT DEFAULT CURRENT_TIMESTAMP,

    FOREIGN KEY (project_id) REFERENCES projects(id),
    FOREIGN KEY (icp_id) REFERENCES icps(id)
);

12.7 Optional evidence table

Better version:

CREATE TABLE knowledge_evidence (
    id INTEGER PRIMARY KEY AUTOINCREMENT,

    knowledge_item_id INTEGER NOT NULL,

    source_type TEXT NOT NULL,
    source_id TEXT,
    source_url TEXT,

    evidence_text TEXT,
    raw_data TEXT,

Use this when multiple replies or calls support the same insight.

ICP refinement

Signal extraction

Knowledge tells signal extraction:

which patterns are meaningful
which competitors matter
which phrases indicate pain
Scoring

Knowledge modifies scoring:

companies with historically strong signals get boosted
personas with poor response rates get lower priority
bad-fit patterns are penalized
Draft generation

Knowledge improves copy:

use winning phrases
avoid failed angles
handle common objections
adapt messaging by persona
Quality control

Knowledge adds rules:

avoid claims that caused negative replies
avoid overused subject lines
avoid generic personalization

12.12 Knowledge extraction with AI

Use AI to extract structured insights from raw feedback.

Prompt:

You analyze outbound campaign feedback and extract reusable GTM knowledge.

Given:
- ICP definition
- campaign reply or transcript
- lead profile
- email draft that was sent
- campaign outcome

Return JSON only:
{
  "knowledge_items": [
    {
      "item_type": "",
      "title": "",
      "summary": "",
      "content": "",
      "tags": [],
      "applies_to_persona": "",
      "applies_to_segment": "",
      "applies_to_competitor": "",
      "importance_score": 0.0,
      "confidence_score": 0.0,
      "evidence_text": ""
    }
  ]
}

Rules:
- Extract only reusable insights.
- Do not create knowledge from one weak ambiguous reply unless marked low confidence.
- Keep titles short.
- Evidence text must come from source content.
- If nothing useful exists, return {"knowledge_items": []}.

12.14 Knowledge deduplication

Avoid storing the same insight many times.

Deduplicate by:

item_type
normalized title
applies_to_persona
applies_to_competitor
applies_to_segment

If same insight appears again:

increase evidence_count
increase confidence_score
update summary if better
add evidence
12.15 Confidence scoring

Confidence should depend on evidence.

Example:

manual note from user: 0.8
1 positive reply: 0.4
3 similar replies: 0.7
10 similar replies: 0.9
demo call evidence: 0.8
multiple source types: +0.1

Cap at:

0.95
12.16 Importance scoring

Importance answers:

How much should this affect future outreach?

Factors:

appears in positive replies
appears across multiple leads
linked to meetings booked
linked to negative replies
high-value segment
high-confidence source

Example:

Objection causing many rejections → high importance
One random comment → low importance
12.17 Knowledge lifecycle

Statuses:

active
needs_review
archived
rejected
stale

Mark as stale when:

old campaign pattern no longer works
ICP changed significantly
new evidence contradicts it
12.18 Contradiction handling

The system should allow conflicting knowledge.

Example:

Founders respond well to cost-saving angle.
CRM managers respond poorly to cost-saving angle.

This is not a contradiction if persona differs.

But this is:

“Cost-saving angle works for CRM managers”
“Cost-saving angle performs poorly for CRM managers”

Mark both as:

needs_review

or store confidence separately based on newer evidence.

12.20 MVP retrieval without vector DB

Since you only have SQLite:

Use:

structured filters
tags
LIKE search
importance sorting

Example:

SELECT *
FROM knowledge_items
WHERE project_id = ?
  AND icp_id = ?
  AND status = 'active'
  AND (
    applies_to_persona = ?
    OR applies_to_competitor = ?
    OR tags LIKE ?
  )
ORDER BY importance_score DESC, confidence_score DESC
LIMIT 10;

Later:

add embeddings + vector DB
or SQLite FTS5

12.24 Main module pseudocode
class KnowledgeBaseModule(PipelineModule):
    name = "knowledge_base"

    def run(self, context):
        sources = self.load_sources(context)

        created = 0
        updated = 0
        failed = 0

        for source in sources:
            try:
                extracted_items = extraction_service.extract(source, context)

                for item in extracted_items:
                    existing = dedupe_service.find_similar(item)

                    if existing:
                        knowledge_repo.update_with_evidence(existing.id, item)
                        updated += 1
                    else:
                        knowledge_repo.create(item)
                        created += 1

            except Exception as e:
                failed += 1
                log_error(e, source)

        return ModuleResult(
            success=True,
            input_count=len(sources),
            output_count=created + updated,
            failed_count=failed
        )

12.28 How draft generation should use it

Before generating an email draft, retrieve:

winning messages for this ICP
objections for this persona
competitor notes for detected competitor
pain points for this segment
draft rules

Example:

Lead uses Klaviyo.
Contact is Head of CRM.

Retrieve:
- Klaviyo objections
- CRM leader pain points
- winning messages for ecommerce lifecycle automation
12.29 How scoring should use it

Knowledge can adjust scores.

Examples:

If ICP segment has high positive reply rate → boost
If persona usually rejects → lower persona priority
If competitor displacement signal works well → boost signal
If repeated objection exists → needs better copy, not rejection

MVP:

do not modify scores automatically
only show knowledge in summary

Later:

use knowledge-based score modifiers
12.30 How ICP refinement should use it

Knowledge should create suggestions like:

Add “Klaviyo migration risk” as objection
Increase priority for CRM Manager persona
Lower priority for generic Marketing Manager
Add “hiring lifecycle marketer” as strong signal
Exclude agencies

Store suggestions as:

item_type = icp_refinement

Do not auto-apply in MVP.

12.34 Definition of done

This module is complete when:

You can store campaign learnings
You can add manual GTM notes
Replies can be analyzed into reusable insights
Insights are deduplicated
Knowledge has confidence and importance scores
Knowledge can be retrieved by ICP/persona/competitor
Draft generation can use relevant knowledge
ICP refinement can receive suggestions
Old or bad knowledge can be archived
