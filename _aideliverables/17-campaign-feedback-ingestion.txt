17. Campaign feedback ingestion module
17.1 Purpose

Import campaign outcomes back into the system so it can learn.

This module turns:

sent emails, replies, bounces, unsubscribes, booked calls

into structured feedback used by:

suppression
knowledge base
ICP refinement
scoring
future draft generation
17.2 Main goal

After outreach happens, the system should know:

Who was contacted?
Who replied?
Was the reply positive, negative, neutral?
What objections came up?
Which angle worked?
Which leads should never be contacted again?
What should change in the next campaign?

17.6 Campaign event table

Current table is good:

CREATE TABLE campaign_events (
    id INTEGER PRIMARY KEY AUTOINCREMENT,

    campaign_id INTEGER,
    lead_candidate_id INTEGER,
    contact_id INTEGER,

    event_type TEXT NOT NULL,
    event_value TEXT,

    reply_text TEXT,
    reply_sentiment TEXT,
    reply_intent TEXT,

    raw_data TEXT,

    occurred_at TEXT,
    created_at TEXT DEFAULT CURRENT_TIMESTAMP,

    FOREIGN KEY (campaign_id) REFERENCES campaigns(id),
    FOREIGN KEY (lead_candidate_id) REFERENCES lead_candidates(id),
    FOREIGN KEY (contact_id) REFERENCES contacts(id)
);

Recommended additions:

ALTER TABLE campaign_events ADD COLUMN email TEXT;
ALTER TABLE campaign_events ADD COLUMN external_event_id TEXT;
ALTER TABLE campaign_events ADD COLUMN external_campaign_id TEXT;
ALTER TABLE campaign_events ADD COLUMN provider TEXT;
ALTER TABLE campaign_events ADD COLUMN processed INTEGER DEFAULT 0;

Recommended unique index:

CREATE UNIQUE INDEX idx_unique_campaign_event
ON campaign_events(provider, external_event_id);

For CSV imports where no event ID exists, create a hash:

provider + email + event_type + occurred_at + reply_text_hash
17.7 Campaign table
CREATE TABLE campaigns (
    id INTEGER PRIMARY KEY AUTOINCREMENT,

    project_id INTEGER NOT NULL,
    icp_id INTEGER,

    name TEXT NOT NULL,
    channel TEXT DEFAULT 'email',

    provider TEXT,
    external_campaign_id TEXT,

    status TEXT DEFAULT 'active',

    created_at TEXT DEFAULT CURRENT_TIMESTAMP,
    updated_at TEXT DEFAULT CURRENT_TIMESTAMP,

    FOREIGN KEY (project_id) REFERENCES projects(id),
    FOREIGN KEY (icp_id) REFERENCES icps(id)
);

Campaign statuses:

draft
active
paused
completed
archived

17.10 Main flow
1. Load campaign or import file
2. Fetch raw feedback events
3. Normalize provider-specific fields
4. Match event to contact/lead
5. Deduplicate event
6. Save campaign event
7. Classify replies
8. Apply suppression for bounces/unsubscribes
9. Update lead/contact statuses
10. Send useful replies to knowledge base
11. Log result

17.12 Matching events to leads

Match in this order:

1. lead_candidate_id if provided
2. email
3. contact_id if provided
4. external system ID if stored
5. company domain + contact name fallback

Best MVP:

match by email

}

17.15 Reply classification prompt
You classify cold outreach replies.

Given:
- original email draft
- lead profile
- reply text

Return JSON only:
{
  "reply_sentiment": "positive|negative|neutral",
  "reply_intent": "",
  "summary": "",
  "objections": [],
  "competitors_mentioned": [],
  "should_suppress": true/false,
  "should_follow_up": true/false,
  "suggested_next_action": ""
}

Rules:
- If they ask to unsubscribe or stop contacting, should_suppress = true.
- If bounce/unsubscribe, do not classify as sales objection.
- Keep summary short.
- Do not invent intent.

add email to suppression_list

17.17 Suppression handling

Automatically suppress:

unsubscribed
spam_complaint
hard bounce
manual remove request

Maybe suppress or cooldown:

not interested
wrong person
not now

Recommended:

not interested → suppress contact
not now → cooldown, not suppress forever
wrong person → suppress contact, not company

17.25 Definition of done

This module is complete when:

Campaign results can be imported
Events are normalized
Events are deduplicated
Events match existing leads/contacts
Replies are classified
Bounces and unsubscribes suppress contacts
Lead/contact statuses are updated
Useful feedback enters the knowledge base
Campaign summaries are generated
Future campaigns can learn from previous outcomes
