3. Modular pipeline architecture
3.1 Goal

Build a modular outreach pipeline where each module:

has one clear responsibility
reads from SQLite
writes back to SQLite
can be replaced later
can fail without breaking the whole system
can be run manually or in the background

Final output:

ready outreach leads + email drafts

3.3 Core design principle

Each module should follow this pattern:

Input → Process → Save → Mark Status → Log Result

No module should directly depend on the internal logic of another module.

It should only depend on:

database records
module config
shared interfaces
3.4 Module interface

Every module should implement the same basic interface.

class PipelineModule:
    name: str

    def run(self, context: PipelineContext) -> ModuleResult:
        pass
PipelineContext
from dataclasses import dataclass
from typing import Optional, Dict, Any

@dataclass
class PipelineContext:
    project_id: int
    icp_id: Optional[int]
    pipeline_run_id: int
    config: Dict[str, Any]
    dry_run: bool = False
ModuleResult
from dataclasses import dataclass
from typing import Optional, Dict, Any

@dataclass
class ModuleResult:
    success: bool
    input_count: int = 0
    output_count: int = 0
    failed_count: int = 0
    message: Optional[str] = None
    data: Optional[Dict[str, Any]] = None
3.5 Pipeline orchestrator

The orchestrator controls which modules run and in what order.

class PipelineOrchestrator:
    def __init__(self, modules):
        self.modules = modules

    def run(self, context: PipelineContext):
        for module in self.modules:
            result = module.run(context)

            if not result.success:
                # log failure
                # decide whether to stop or continue
                break

3.9 Data handoff between modules

Each module should not pass large objects directly to the next one.

Instead:

Module A writes to SQLite
Module B reads from SQLite

Example:

Company Discovery writes companies
Company Enrichment reads companies where status = discovered

This makes the system restartable.

3.10 Module input/output contract

Each module should have a contract file.

class CompanyDiscoveryInput:

3.11 Provider abstraction

Each module can use providers.

Example:

CompanyDiscoveryModule
  → ApifyCompanyDiscoveryProvider
  → GoogleSearchProvider
  → CSVUploadProvider

Provider interface:

class CompanyDiscoveryProvider:
    def discover(self, icp, config) -> list[dict]:
        pass

This allows you to replace Apify later without rewriting the module.

3.14 Error handling

Each module should handle item-level failures.

Bad:

1 failed company kills the whole run

Good:

1 failed company is logged, pipeline continues

Recommended behavior:

module fails only if provider/system failure happens
individual item failures are counted and logged
3.15 Retry strategy

Retry only external operations.

Examples:

API timeout
rate limit
temporary scraper error
AI API error

Do not retry:

invalid URL
missing domain
invalid email
suppressed contact

Simple retry config:

MAX_RETRIES = 3
RETRY_BACKOFF_SECONDS = [5, 20, 60]
3.16 Idempotency rules

Every module must be safe to rerun.

Examples:

Company discovery:

If domain already exists, update source, do not create duplicate.

Contact discovery:

If same email exists, reuse contact.

Draft generation:

If approved draft exists, do not overwrite.
If draft failed QC, generate new version.

Signal extraction:

If same signal_type + company_id + source_url exists, update it instead of duplicating.
3.17 Background runner

MVP can use a simple Python loop.

while True:
    find active ICPs
    run selected pipeline
    sleep

Better MVP:

manual trigger from UI
plus scheduled daily run

Suggested schedule:

company discovery: daily
company enrichment: hourly
contact enrichment: hourly
draft generation: daily
feedback ingestion: daily
3.18 Queue model for SQLite

SQLite does not need a complex queue at first.

Use status fields:

companies.status = discovered
contacts.status = new
lead_candidates.lead_status = qualified

Each module picks records by status.

Example:

SELECT *
FROM companies
WHERE status = 'discovered'
LIMIT 50;

Then mark as:

processing
enriched
failed
3.19 Concurrency rule

For SQLite MVP:

single worker only

Avoid parallel writes until needed.

Later:

multiple workers
PostgreSQL
real queue
Redis/RQ/Celery
